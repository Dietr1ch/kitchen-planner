#+title: Kitchen Planner

A CLI tool for modelling kitchen resources, recipes, and cooks, then
generating optimized cooking schedules via constraint programming.

* Quick Start

Generate a meal plan and render it:

#+begin_src sh
  nix develop
  just plan        # solve + render text Gantt
  just run-html /tmp/plan.html   # render as HTML
#+end_src

* Usage

Two CLI tools, piped together:

#+begin_src sh
  # Solve: kitchen + cooks + recipes → Plan JSON (stdout)
  cargo run \
  	--bin schedule \
  	-- \
  	schedule \
  	<kitchen.ron> \
  	--cook <cook..> \
  	<recipes...>

  # Render: Plan JSON → Gantt chart (text/html)
  cargo run \
  	--bin gantt \
  	-- \
  	[--format text|html] \
  	<plan.json>
#+end_src

Example:

#+begin_src sh
  cargo run \
  	--bin schedule \
  	-- \
  	schedule \
  	data/kitchens/simple.ron \
  	--cook data/cooks/alice.ron \
  	data/recipes/lasagna.ron \
  	| cargo run \
  	    --bin gantt \
  	    -- \
  	    -
#+end_src

* Data Model

** Kitchen (~data/kitchens/*.ron~)

- =Equipment= — =id=, =name=, =kind= (=burner=, =oven=, =pot=, =prep_surface=)
- =Food= — =id=, =name=, =quantity=, =unit=
- =Materials= — =id=, =name=, =quantity=, =unit=

Equipment =kind= is used by the solver to match tasks to available
equipment. Whether a task occupies a cook is controlled by the
=needs_cook= field on each recipe step (defaults to =true=; set to
=false= for oven, rest, thaw, or cooling steps).

** Recipe (~data/recipes/*.ron~)

- =name=, =ingredients=, =steps=
- Each =Step=: =id=, =description=, =duration_minutes=, =resource_kind=,
  =dependencies= (step IDs within same recipe), =optional=, =needs_cook=,
  =skill=, =min_skill_level=, =duration_by_skill=

Resource references equipment by =kind= from the kitchen (e.g. ="oven"=,
="burner"=, ="prep_surface"=; =null= means no equipment needed). The solver
assigns each task to a specific piece of equipment matching the required kind.
Dependencies are resolved by prepending =recipe_name:= to each step ID
(e.g., =s1= → =Lasagna:s1=).

** Cook (~data/cooks/*.ron~)

One RON file per cook. =name=, =skills= (map of skill name to =SkillLevel=:
=Unskilled=, =Novice=, =Intermediate=, =Advanced=, =Expert=). The solver
uses skills to determine effective task durations: if a step defines
=duration_by_skill=, the solver picks the best available duration matching
the assigned cook's skill level, falling back to the nearest lower level.

** Plan (solver output)

- =start_time= (e.g. ="18:00"=)
- =Tasks=: =id=, =description=, =start_offset_minutes=,
  =duration_minutes=, =resource_kind=, =resource_id=, =cook=, =dependencies=

* Solver (=schedule= binary)

Uses MiniZinc with Gecode to produce an optimized schedule.

** Model

- Decision variables: =start[t]= (0..horizon), =cook[t]= (0..num_cooks),
  =assign[t]= (0..num_equipment)
- Constraints:
  - Dependency chains: =start[child] >= start[parent] + duration[parent]=
  - Resource disjunction: same-equipment tasks cannot overlap
  - Equipment kind matching: each task's =assign[t]= must map to a kitchen
    equipment whose =kind= matches the task's =resource_kind=
  - Cook disjunction: same-cook tasks cannot overlap (=oven= tasks exempt)
  - Skill eligibility: tasks requiring a skill are only assigned to cooks
    whose skill level meets =min_skill_level=
- Objective: =minimize max_end * (horizon + 1) + (max_end - min_recipe_end)=
  — weighted combination of makespan and recipe end-time alignment

** Cook allocation

- Tasks with =needs_cook= =false= (oven, rest, thaw, cooling) → =cook= 0 (no cook assigned)
- Tasks with =needs_cook= =true= → assigned a distinct cook (1..num_cooks), blocking that
  cook for the duration
- Tasks requiring a skill (=skill= / =min_skill_level=) only assigned to cooks
  whose level meets or exceeds the requirement; effective duration is looked up
  from =duration_by_skill=, falling back to the nearest lower level

* Renderer (=gantt= binary)

** Text Renderer (~src/render/text.rs~)

Terminal Gantt chart with proportional bars. Shows time range inside the
duration column. Left offset is proportional to start time.

** HTML Renderer (~src/render/html.rs~)

Interactive HTML Gantt chart. Hovering a row highlights its upstream
dependencies (orange) and downstream dependents (blue). Bar position
and width are percentage-based via CSS =margin-left=/=width=.

*** Self-contained Web App (~assets/gantt.html~)

Standalone client-side Gantt chart viewer. Load a plan JSON file via
drag-and-drop, file picker, or a =?url= query parameter. Implemented in
pure HTML/CSS/JS with no external dependencies. Served locally via
=static-web-server=:

#+begin_src sh
  just serve
#+end_src

* Conventions & Decisions

** Code Conventions

- Rust edition 2024, stable toolchain (1.75+ from ~rust-toolchain.toml~)
- ~serde~ with ~derive~ for all JSON schema types
- ~clap~ with ~derive~ for CLI argument parsing (self-documenting)
- All file path arguments accept ~-~ as alias for stdin (resolved to ~/dev/stdin~)
- Output uses ~write!~ + ~flush~ instead of ~print!~ to avoid SIGPIPE panics
- No library dependencies beyond ~serde~, ~serde_json~, ~ron~, ~clap~, ~strum~
- Nix/Flakes for reproducible builds; ~shell.nix~ provides ~minizinc~ + ~gecode~

** Architecture Decisions

- [[https://github.com/anomalyco/opencode][opencode]]-driven development — the project was built entirely via AI-assisted pair programming
- Replaced an initial ad-hoc sequential heuristic with MiniZinc constraint programming
  (the CP solver produces tighter schedules by exploring interleavings a greedy
  heuristic misses; e.g., 100 min vs 110 min makespan for 4 cooks + both recipes)
- Weighted single scalar objective ~max_end * (horizon + 1) + (max_end - min_recipe_end)~
  instead of MiniZinc's lexicographic ~++~ operator (which caused type errors with
  ~var int~ on MiniZinc 2.9.3/Gecode)
- One-indexed arrays in the MiniZinc model (native default) to avoid ~array1d~ coercion errors
- Cook is NOT blocked during oven/rest tasks (=needs_cook= on each step):
  physically accurate (set-and-forget), enables natural interleaving (e.g., GB prep
  during LS bake)
- Cooks are passed via ~--cook~ flags (~data/cooks/~)

** Data Conventions

- Task IDs use ~{recipe_name}:{step_id}~ prefix (e.g., ~Lasagna:s1~); dependencies
  reference these prefixed IDs
- Step dependencies within a recipe file use un-prefixed IDs; the solver prepends
  the recipe name at build time
- ~data-depends-on~ HTML attribute uses ~, ~ (comma-space) as separator to
  handle recipe names containing spaces
- ~resource_kind~ on recipe steps uses equipment ~kind~ (e.g. ~oven~, ~burner~,
  ~prep_surface~); the solver assigns each task to a specific equipment instance
  matching that kind
- ~needs_cook~ on recipe steps (defaults to ~true~) controls whether the step
  occupies a cook. Set to ~false~ for oven baking, resting, thawing, cooling
  — any step that doesn't need active cook attention
- ~skill~ / ~min_skill_level~ / ~duration_by_skill~ on recipe steps enable
  skill-adaptive durations; the solver assigns tasks to eligible cooks and
  picks the best available duration for their skill level
- ~data-start~ and ~data-cook~ HTML attributes on Gantt rows enable
  client-side column sorting by start time or cook name

** Renderer Decisions

- Time range shown inside the duration bar column (not a separate column):
  saves horizontal space and keeps the time label physically on the bar
- Text renderer uses ~▓~ characters with left offset proportional to start time;
  bars that would overflow the column width are truncated (time label takes priority)
- HTML renderer uses percentage-based ~margin-left~/~width~ with ~overflow: hidden~
  on the bar container to prevent bars spilling into adjacent columns
- Interactive hover highlights upstream dependencies (orange) and downstream
  dependents (blue) via data attributes (~data-task-id~ / ~data-depends-on~)
- Sort order can be by start time or by cook name via ~gantt --sort-by start|cook~
  (client-side in the web app via column header clicks)
- Two HTML rendering paths: Rust binary (~gantt --format html~) generates
  static HTML at build time; client-side web app (~assets/gantt.html~) loads
  and renders plan JSON at runtime via drag-and-drop or =?url=

* Status

Completed schemas for kitchen, recipe, cook, and plan (JSON via ~serde~).

MiniZinc-based solver produces optimized schedules with:
- Dependency chain enforcement
- Resource contention avoidance (same equipment)
- Cook contention avoidance (oven tasks exempt)
- Recipe end-time alignment via weighted objective
- Skill-adaptive durations: cooks are assigned to tasks matching their skill
  level, and effective durations are picked from =duration_by_skill=

Three renderers (text terminal, Rust HTML, client-side web app).
